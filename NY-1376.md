# NY-1376: Flash when opening the automation editor

## Hypothesis

The flash is caused by a one-render gap between the automation read query resolving and the editor seeding its local `draft` state.

`AutomationEditor` receives the loaded automation from `useReadAutomation`, but it passes `draft` to `AutomationCanvas`. `draft` starts as `undefined` and is populated from the loaded automation in a `useEffect`, which runs after render. During that intermediate render, the query is no longer loading and has not errored, but `AutomationCanvas` receives no automation and renders the `Couldn't load automation` banner.

## Experiment Results

Temporary logs were added to `AutomationEditor` and `AutomationCanvas`.

On a hard refresh that reproduced the flash, the editor rendered in this sequence:

```js
{
    id: '6a3bdefd52a7dc0024a7546d',
    isLoadingAutomation: true,
    isError: false,
    hasLoadedAutomation: false,
    hasDraft: false,
    automationId: undefined,
    draftId: undefined
}
```

Then, after the read query resolved:

```js
{
    id: '6a3bdefd52a7dc0024a7546d',
    isLoadingAutomation: false,
    isError: false,
    hasLoadedAutomation: true,
    hasDraft: false,
    automationId: '6a3bdefd52a7dc0024a7546d',
    draftId: undefined
}
```

During that same frame, the canvas rendered the error branch:

```js
{
    isLoading: false,
    isError: false,
    hasAutomation: false,
    hasGraph: false,
    automationId: undefined,
    actionCount: undefined,
    edgeCount: undefined
}
```

Immediately after the effect seeded `draft`, the editor rendered successfully:

```js
{
    id: '6a3bdefd52a7dc0024a7546d',
    isLoadingAutomation: false,
    isError: false,
    hasLoadedAutomation: true,
    hasDraft: true,
    automationId: '6a3bdefd52a7dc0024a7546d',
    draftId: '6a3bdefd52a7dc0024a7546d'
}
```

## Root Cause Analysis

The query state is not the root cause. `useReadAutomation` correctly reports that loading is complete and the automation is available.

The root cause is local editor state synchronization:

1. `AutomationEditor` derives `automation` from `useReadAutomation`.
2. `draft` is initialized as `undefined`.
3. `draft` is seeded from `automation` inside `useEffect`.
4. `AutomationCanvas` is rendered with `automation={draft}`.
5. On the first render after the query resolves, `automation` exists but `draft` is still `undefined`.
6. `AutomationCanvas` treats `!automation` as a load failure and renders `Couldn't load automation`.
7. The effect runs, sets `draft`, and the editor renders normally.

This explains the brief flash with no failed network request and no console error. The error banner is not representing a failed automation load; it is representing a transient uninitialized local draft.

## Fix

Keep rendering the editor from the local `draft`, but treat the frame where the server automation has loaded and the draft has not yet been seeded as an editor initialization state.

The editor now derives:

```ts
const isInitializingDraft = !!automation && !draft;
const isEditorLoading = isLoadingAutomation || isInitializingDraft;
```

`AutomationHeader` and `AutomationCanvas` still receive `draft`, preserving the existing model that the editor UI is driven by the editable local copy. During the one render before `useEffect` seeds `draft`, they receive `isEditorLoading`, so the canvas renders the loading state instead of misclassifying the missing draft as a failed automation load.

The fix is intentionally local to `AutomationEditor` because `useReadAutomation` is already returning the correct query state. The bug was not a read-query loading problem; it was that the editor treated an uninitialized local draft as if the automation itself failed to load.

The regression test renders the editor with automation data already available and asserts that the load-error banner is not present on the first committed render.
